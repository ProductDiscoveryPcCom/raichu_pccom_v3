# Auditoria de Salud del Proyecto — Raichu v5.1.0

**Fecha:** 2026-03-22
**Autor:** Equipo Product Discovery & Content
**Scope:** Mantenibilidad, consistencia, cobertura de tests, dependencias

## Resumen ejecutivo

Raichu v5.1.0 esta funcional y desplegado en Streamlit Cloud. Sin embargo, la iteracion rapida ha acumulado deuda tecnica significativa: el sistema de versiones esta fragmentado en 6 ubicaciones con valores distintos, el entrypoint es un monolito de 1592 lineas, y la suite de tests proporciona cobertura practicamente nula. A continuacion se documentan 8 hallazgos priorizados por impacto y esfuerzo.

## Detalle de hallazgos

### F1 — Version inconsistente en 6 ubicaciones

**Severidad:** Alta | **Impacto:** Alto | **Esfuerzo:** Bajo

**Estado actual:**
- `VERSION` (archivo raiz) = **"5.1.0"** (correcto, source of truth)
- `version.py` L18 fallback = **"5.1.0"** (correcto)
- `app.py` docstring L3 = **"Version 1.2.0"** (incorrecto)
- `config/settings.py` L13 `__version__` = **"1.2.0"** (incorrecto)
- `config/arquetipos.py` L38 `__version__` = **"1.3.0"** (incorrecto)
- `CHANGELOG.md` ultima entrada = **"[1.2.0] - 2026-02-10"** (desactualizado)

**Riesgo:** Confusion sobre que version esta desplegada. El debug panel muestra "5.1.0" (correcto), pero un desarrollador que lea el codigo fuente vera "1.2.0" en multiples sitios.

**Accion recomendada:**
1. Eliminar `__version__` de `config/settings.py` y `config/arquetipos.py`
2. Actualizar docstring de `app.py` L3 a la version actual
3. Actualizar `CHANGELOG.md` con entradas desde 1.2.0 hasta 5.1.0 (o al menos marcar el salto)
4. Establecer regla: solo el archivo `VERSION` es autoritativo, todo el codigo importa de `version.py`

### F2 — app.py monolito (1592 lineas, 29 funciones)

**Severidad:** Alta | **Impacto:** Alto | **Esfuerzo:** Alto

**Estado actual:**
`app.py` contiene 29 funciones que mezclan responsabilidades:
- Autenticacion (`check_auth`, 50 lineas)
- Carga de config y bridge de API keys (143 lineas, L72-214)
- Gestion de session state (inicializacion, save/restore modos, clear)
- 5 renderers de modo (new, rewrite, verify, assistant + guides)
- Delegacion de pipeline y post-procesamiento de resultados
- Footer y panel de debug
- Definiciones fallback para cuando los imports fallan

**Riesgo:** Cada cambio en cualquier area puede causar efectos laterales. Imposible testear funciones en aislamiento. Dificil para nuevos desarrolladores orientarse.

**Accion recomendada:**
Extraccion progresiva siguiendo el plan en `.claude/agents/refactor-agent.md`:
1. Extraer `auth.py` (bajo riesgo, autocontenido)
2. Extraer `app_config.py` (bajo riesgo, carga de config + bridge)
3. Extraer `state.py` (medio riesgo, gestion de session state)
4. Extraer `ui/verify.py` (medio riesgo, consistente con ui/opportunities.py existente)

Objetivo: reducir app.py a ~400 lineas (imports, routing, header, fallbacks).

**Prerequisito:** F3 (tener tests antes de refactorizar).

### F3 — Suite de tests minima

**Severidad:** Alta | **Impacto:** Alto | **Esfuerzo:** Medio

**Estado actual:**
- `tests/test_modular.py`: 37 lineas, 3 tests basicos (imports, arquetipos count, word count)
- 6 archivos de test adicionales (`test_fixes.py` 920 lineas, `test_ux_ui_functionality.py` 1171 lineas, etc.) generados en bulk — probablemente no pasan sin entorno Streamlit completo
- Zero cobertura en: prompt builders, pipeline, validacion de inputs, session state, correccion dual

**Riesgo:** Regresiones pasan desapercibidas. El refactoring de F2 es peligroso sin tests que validen el comportamiento existente.

**Accion recomendada:**
1. Verificar que tests existentes pasan: `pytest tests/test_modular.py -v`
2. Escribir unit tests para funciones puras prioritarias:
   - Prompt builders: mock de inputs, assert que output contiene estructura HTML CMS obligatoria
   - `config/arquetipos.py`: validar esquema de cada arquetipo (code, name, tone, structure, lengths)
   - `utils/html_utils.py`: word count, validate_html_structure, detect_ai_phrases
   - Session state isolation: mock de st.session_state, verificar save/restore
3. Objetivo: 60% cobertura en `prompts/` y `utils/`

### F4 — Dependencias no usadas en requirements

**Severidad:** Media | **Impacto:** Medio | **Esfuerzo:** Bajo

**Estado actual:**
Dependencias en `requirements.txt` sin ninguna importacion en el codebase:
- **PyYAML** — comentario "para futura externalizacion de prompts", pero los prompts son funciones Python
- **Jinja2** — mismo comentario, sin uso
- **openpyxl** — marcado como opcional, sin importacion encontrada
- **python-docx** — marcado como opcional, sin importacion encontrada

En `requirements-dev.txt`:
- **jupyter, ipython** — sin notebooks en el repo
- **mkdocs, mkdocs-material** — sin sitio de documentacion

**Riesgo:** Installs mas lentos, footprint de deploy mayor, superficie de vulnerabilidades ampliada innecesariamente.

**Accion recomendada:**
1. Verificar con `grep -r "import yaml\|import jinja2\|import openpyxl\|import docx"` (ya verificado: 0 resultados)
2. Eliminar PyYAML, Jinja2, openpyxl, python-docx de `requirements.txt`
3. Eliminar jupyter, ipython, mkdocs, mkdocs-material de `requirements-dev.txt`
4. Re-deploy y verificar que la app funciona

**Riesgo:** Bajo — el codigo funciona, solo hay discrepancia con la documentacion.

**Accion recomendada:**
Resolver como parte de F2: al extraer la logica de state a un modulo dedicado, crearlo como `state.py` o `utils/state_manager.py`.

### F6 — Changelog embebido en docstring de app.py

**Severidad:** Media | **Impacto:** Bajo | **Esfuerzo:** Bajo

**Estado actual:**
Las lineas 1-33 de `app.py` contienen un docstring con changelog que cubre versiones 4.5.0 a 4.9.2, pero la cabecera dice "Version 1.2.0". Esta informacion:
- Duplica y contradice `CHANGELOG.md`
- Hace el docstring ilegible (33 lineas de changelog antes de llegar al codigo)
- Referencia versiones que no coinciden con el source of truth (VERSION = 5.1.0)

**Riesgo:** Confunde a desarrolladores que lean el archivo. Informacion contradictoria.

**Accion recomendada:**
Reemplazar las 33 lineas por un docstring conciso:
```python
"""
PcComponentes Content Generator - App Principal
Aplicacion Streamlit para generacion de contenido SEO.
Flujo de 3 etapas: Borrador -> Analisis -> Final
"""
```
Toda la informacion de changelog pertenece a `CHANGELOG.md`.

### F7 — API keys copiadas a os.environ (bridge pattern)

**Severidad:** Media | **Impacto:** Medio | **Esfuerzo:** Medio

**Estado actual:**
`app.py` L150-214 copia API keys de `st.secrets` a `os.environ` para que modulos downstream (`core/generator.py`, `config/settings.py`, `utils/serp_research.py`) las encuentren via `os.getenv()`.

```python
if CLAUDE_API_KEY:
    os.environ['ANTHROPIC_API_KEY'] = CLAUDE_API_KEY
```

**Riesgo:** Las keys quedan visibles en el entorno del proceso. En Streamlit Cloud esto esta contenido, pero en desarrollo local o si alguna dependencia loguea variables de entorno, las keys podrian filtrarse. Ademas, es un acoplamiento fragil: si un modulo espera un nombre de env var diferente, falla silenciosamente.

**Accion recomendada:**
- `ContentGenerator` ya acepta `api_key` en su constructor — usarlo directamente en lugar del bridge para ANTHROPIC_API_KEY
- Evaluar si SEMrush y SerpAPI realmente requieren env vars o tambien aceptan parametros
- Prerequisito: F2 (refactorizar app_config.py para centralizar la inyeccion)

---

### F8 — Sin CI pipeline

**Severidad:** Media | **Impacto:** Medio | **Esfuerzo:** Medio

**Estado actual:**
`requirements-dev.txt` incluye herramientas de calidad: pytest, black, flake8, mypy, bandit, safety, pre-commit. Pero no existen:
- `.github/workflows/` — sin GitHub Actions
- `.pre-commit-config.yaml` — sin hooks de pre-commit
- `pyproject.toml` con config de herramientas — sin configuracion centralizada
- `Makefile` o scripts de CI — sin shortcuts

Las herramientas estan instalables pero nadie las ejecuta automaticamente.

**Riesgo:** Problemas de estilo, tipos y seguridad no se detectan hasta que causan bugs en produccion.

**Accion recomendada:**
1. Crear `.github/workflows/ci.yml` con: `black --check`, `flake8`, `pytest tests/`, `safety check`
2. Crear `.pre-commit-config.yaml` con hooks de black + flake8
3. Configurar herramientas en `pyproject.toml` (line-length, excludes, etc.)
